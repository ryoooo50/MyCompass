# My Compass 要件定義書

- 文書バージョン: 0.1.0
- 作成日: 2026-06-22
- 対象: 現在の `index.html` ベースのプロトタイプ
- 文書の目的: 現在の実装を正として、システムの目的・仕様・制約・今後の要件を明確にする

## 1. システム概要

My Compass は、個人の生活・研究・仕事に関する情報を一画面で把握するための個人用ダッシュボードである。

財務、タスク、研究、アルバイト、予定、外部アプリへの導線をまとめ、利用者が「今日やること」と各領域の進捗を短時間で確認できる状態を目指す。

現段階は画面設計と操作感を検証するためのフロントエンドプロトタイプであり、表示データの大半はモックデータである。

## 2. 背景と目的

### 2.1 背景

個人の情報が Obsidian、Notion、Google Calendar、Google Drive、GitHub など複数の場所に分散しているため、現在地や次の行動を把握するまでに手間がかかる。

### 2.2 目的

- 生活・研究・仕事の状態を一か所で俯瞰できるようにする
- 今日取り組む項目を明確にする
- よく使う外部サービスへ素早く移動できるようにする
- 将来のデータ連携に向けて、先に情報設計と画面設計を検証する

### 2.3 現段階で対象外とすること

- 実データを用いた本番運用
- 外部サービスとのAPI同期
- 複数利用者での利用
- クラウド上での認証・認可
- サーバー側でのデータ永続化

## 3. 想定利用者

- 主利用者: この環境を所有する単一ユーザー
- 利用端末: PC、タブレット、スマートフォンのWebブラウザ
- 想定利用頻度: 日次での状況確認、タスク完了操作、外部アプリへの移動

## 4. 用語

| 用語 | 定義 |
| --- | --- |
| ホーム | 各領域の指標、今日のタスク、研究進捗、予定等をまとめて表示する画面 |
| ビュー | サイドナビゲーションから切り替える画面単位 |
| モックデータ | UI確認用にHTMLへ固定記述されている架空の表示データ |
| 端末内保存 | ブラウザの `localStorage` への保存 |
| 接続設定 | Obsidian Vault名、Notion URL、追加アプリURL等の設定 |

## 6. データ要件

### 6.1 現在保存するデータ

`localStorage` のキー `myCompass` に、JSON形式で次の値を保存する。

| 項目 | 型 | 内容 |
| --- | --- | --- |
| `doneTasks` | 文字列配列 | 完了状態にあるモックタスクのID |
| `vaultName` | 文字列 | Obsidian Vault名 |
| `notionUrl` | 文字列 | NotionホームURL |
| `customScheme` | 文字列 | 追加アプリ用のURLまたはURLスキーム |

### 6.2 データ方針

- 現段階では外部サービスへデータを送信しない
- 表示される金額、予定、研究進捗、勤務時間等は実データではない
- APIキー、アクセストークン、パスワード等の秘密情報をフロントエンドまたは `localStorage` に保存しない
- 将来Notion等と同期する場合、秘密情報はローカルバックエンド等の安全な実行環境で管理する
- データ構造を変更する際は、既存の端末内データを失わない移行方法を検討する

## 7. 非機能要件

### 7.1 ユーザビリティ

- ホームを開いて主要な状態を短時間で把握できること
- 主な操作結果が画面上ですぐ確認できること
- 未実装機能は動作するように見せかけず、未実装であることを明示すること

### 7.2 レスポンシブ対応

- 1000px以下ではサイドバーを縮小し、主要カードの配置を調整すること
- 760px以下ではモバイルメニューへ切り替えること
- 480px以下では指標カードを1列表示にすること
- 週間予定は狭い画面で横スクロールできること

### 7.3 セキュリティ・プライバシー

- 秘密情報をHTML、JavaScript、Git管理対象ファイルへ直接記述しないこと
- 外部リンクを新しいタブで開く場合は `rel="noreferrer"` 等を利用すること
- URL入力値を将来画面へ出力する際は、スクリプト実行や不正なURLスキームを防ぐ検証を行うこと
- 実データへ移行する前に、バックアップ、削除、データ移行の方針を定義すること

### 7.4 保守性

- 現在1ファイルに同居するHTML、CSS、JavaScript、モックデータを、機能拡張前に責務ごとに分離すること
- 仕様変更時は本書を更新すること
- 重要な技術・設計判断は `docs/adr/` に記録すること
- 作業完了時は `developmentLog/` に開発ログを記録すること

### 7.5 アクセシビリティ

- 操作要素はキーボードで到達・操作できること
- アイコンだけのボタンには用途を示すラベルを付けること
- 色だけに依存せず、文字や状態表現でも内容を判別できること
- モーダル実装を拡張する際はフォーカス移動とフォーカストラップを追加すること

## 9. 制約・既知の課題

- 主要な表示値はHTML内に固定されている
- タスク件数の母数がコード上で `6` に固定されている
- `localStorage` の破損したJSONを読み込むと初期化処理が失敗する可能性がある
- 任意アプリURLは保存されるが、画面上で利用されない
- URLの形式検証がない
- モーダルのフォーカス管理が不十分である
- 各詳細ビューはプレースホルダーである
- 自動テスト、静的解析、CIがない
- データのバックアップ・復元手段がない

## 10. 次期開発の優先候補

優先順位は今後の意思決定で確定し、確定時には必要に応じてADRを作成する。

1. HTML、CSS、JavaScript、モックデータの分離
2. データモデルと保存形式の定義
3. タスクの追加・編集・削除
4. 入力検証、エラー処理、アクセシビリティ改善
5. バックアップ用のJSONまたはCSV入出力
6. 財務、研究、勤務、予定の順次実データ化
7. 外部サービス連携用ローカルバックエンドの検討
8. テスト、静的解析、CIの導入

## 11. 現プロトタイプの受入条件

- 静的HTTPサーバーで `index.html` を開き、重大なJavaScriptエラーなくホームを表示できる
- 全7ビューをナビゲーションから切り替えられる
- タスクの完了状態を変更し、再読込後も状態が維持される
- 接続設定を保存し、再読込後もObsidianおよびNotionのリンクへ反映される
- PC、タブレット、スマートフォン相当の幅で主要情報を閲覧できる
- 外部サービスへ個人データや秘密情報を送信しない

## 12. 文書更新ルール

- 実装と本書が食い違う場合、作業内で両者を一致させる
- 実装済み・一部実装・未実装の状態を明確にする
- 新機能では目的、利用者価値、受入条件を追加する
- 技術や仕組みを採用・変更した理由は、必要に応じてADRへ残す
